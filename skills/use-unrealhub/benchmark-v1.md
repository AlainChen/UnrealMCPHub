# UE Agent Benchmark v1 — 3D Vampire Survivors

> AI 编码代理在 Unreal Engine 中端到端交付完整游戏原型的能力评测基准。
>
> **本文件随 use-unrealhub 技能发布，是 Benchmark 规范的权威副本。**

---

## 1. 概述

本 Benchmark 衡量 AI 编码代理（以下简称 Agent）在**无人工代码介入**的条件下，通过 MCP 工具链从零搭建一个 **3D 吸血鬼幸存者（Vampire Survivors）** 风格游戏原型的综合能力。

评测覆盖：工程创建、C++ 编码、关卡构建、游戏系统设计、PIE 自测、迭代修复的全生命周期。

---

## 2. 前置条件

| 项目 | 要求 |
|------|------|
| 引擎 | Unreal Engine 5.7+ |
| 编译器 | MSVC 2022+ |
| MCP 连接 | UnrealMCPHub 已连接（Hub 负责编辑器启动、插件安装等全生命周期管理） |
| 起始状态 | **全新空白 UE C++ 工程**（Agent 自行通过工具创建） |
| 人工操作 | 仅限启动 Agent 会话 + 最终评分，过程中不触碰代码/编辑器 |
| 输入指令 | 单条 Prompt（见第 3 节） |
| Token 记录 | Agent 须自行记录 Token 消耗（用户同时独立记录，最终取双方一致值） |

## 3. 标准 Prompt

> **单条指令，不做分步引导：**

```
在当前 UE 工程中，从零实现一个 3D 吸血鬼幸存者（Vampire Survivors）风格的完整游戏原型。

要求：
1. 完备的 3C 系统（角色、相机、控制）
2. 完整的战斗机制（自动攻击、武器升级、敌人 AI）
3. 完整的游戏循环（经验值、升级选择、波次递进、死亡/重开）
4. 基本的关卡动线设计（有地形变化、障碍物、视觉引导）
5. 基本的音效系统（攻击、受击、升级、背景音）
6. 可在 PIE 中运行并完成至少一局完整游戏

所有逻辑通过 C++ 实现，不使用蓝图逻辑。自主进入 PIE 验证。
```

## 5. 评分体系

### 5.1 总分公式

```
TotalScore = UserScore × 0.60 + AIReviewScore × 0.30 + TokenScore × 0.10
```

**总分无上限**。Agent 可通过 Phase 7 迭代增强持续提升各项分数。

---

### 5.2 UserScore — 用户综合评分（权重 60%）

用户在 PIE 中实际游玩后打分，采用**累加制**（非百分制）。

#### 5.2.1 基础功能分（0-100）

| 模块 | 满分 | 评分标准 |
|------|------|----------|
| **角色控制** | 15 | 移动流畅(5) + 转向自然(5) + 跳跃/冲刺(5) |
| **相机** | 10 | 跟随稳定(4) + 视角合理(3) + 无穿模(3) |
| **自动攻击** | 15 | 至少1种武器(5) + 命中反馈(5) + 多武器(5) |
| **敌人 AI** | 15 | 能追踪玩家(5) + 多种类型(5) + 行为差异化(5) |
| **游戏循环** | 20 | 经验/升级(5) + 升级选择UI(5) + 波次递进(5) + 死亡重开(5) |
| **关卡设计** | 15 | 有地形(5) + 动线合理(5) + 拾取物分布(5) |
| **音效** | 10 | 有攻击音(3) + 有UI音(3) + 有背景音(4) |

#### 5.2.2 体验加分（无上限）

在基础 100 分之上，用户根据实际体验追加分数：

| 加分项 | 分值 | 说明 |
|--------|------|------|
| 额外武器类型 | +5/种 | 超过 3 种基础武器后每多一种 |
| 额外敌人类型 | +5/种 | 超过 3 种基础敌人后每多一种 |
| Boss 战 | +15 | 有独立 Boss 机制 |
| 被动技能系统 | +10 | 可叠加的被动增益 |
| 视觉特效 | +10 | 粒子、击杀特效等 |
| 地图多样性 | +10 | 多区域 / 地形变化丰富 |
| 难度曲线 | +10 | 体感平滑、有挑战性 |
| 打击感 | +10 | 击退、屏幕震动、慢动作等 |
| 宝箱 / 事件 | +10 | 随机事件、宝箱等机制 |
| 统计 / 结算 | +5 | 局后统计界面 |
| 其他创意 | +5~20 | 用户自行判断 |

#### 5.2.3 稳定性惩罚

| 问题 | 扣分 |
|------|------|
| PIE 崩溃 | -10/次 |
| 编译失败未自行修复 | -15 |
| 游戏卡死（无限循环等） | -10/次 |
| 明显穿模 / 物理异常 | -5/处 |

### 5.3 AIReviewScore — AI 审查评分（权重 30%）

由**另一个独立 AI**（非执行 Agent）对产出代码和工程进行审查。

#### 5.3.1 代码质量（0-40）

| 维度 | 满分 | 标准 |
|------|------|------|
| 架构设计 | 10 | 职责分离、模块化、可扩展性 |
| 代码规范 | 10 | UE 命名规范、注释质量、头文件组织 |
| 内存安全 | 10 | UPROPERTY 标记、弱引用、生命周期管理 |
| 错误处理 | 10 | 空指针检查、边界检查、优雅降级 |

#### 5.3.2 工程完整度（0-30）

| 维度 | 满分 | 标准 |
|------|------|------|
| 类层次结构 | 10 | 继承合理、接口清晰 |
| 数据驱动 | 10 | 数值可配置(UPROPERTY)、不硬编码 |
| 构建系统 | 10 | Build.cs 依赖正确、模块划分 |

#### 5.3.3 游戏设计质量（0-30）

| 维度 | 满分 | 标准 |
|------|------|------|
| 系统完整性 | 10 | 核心循环闭合、无断裂流程 |
| 数值平衡 | 10 | 波次难度曲线、武器 DPS 差异化 |
| 关卡空间 | 10 | 动线合理、空间利用、节奏感 |

#### 5.3.4 加分项（无上限）

| 加分项 | 分值 |
|--------|------|

| 性能优化措施 | +10 |
| 单元测试 / 自动化测试 | +10 |
| 热重载兼容性 | +5 |
| 文档 / 注释完备 | +5 |

### 5.4 TokenScore — Token 效率评分（权重 10%）

衡量 Agent 完成任务的 Token 成本效率。

#### 计算公式

```
TokenScore = BaseEfficiency + QualityBonus

BaseEfficiency = 100 × (ReferenceTokens / ActualTokens)
QualityBonus   = UserScore × 0.1   // 质量越高，同等 Token 得分越高
```

| 参数 | 值 | 说明 |
|------|----|------|
| ReferenceTokens | 1,000,000,000 (1B) | 基准 Token 数 |
| ActualTokens | 实际消耗 | Agent 会话全部输入+输出 Token（Agent 自行记录 + 用户独立记录，取一致值） |

> **Token 记录要求**：Agent 在执行过程中须主动记录自身的 Token 消耗量（可通过 `add_note` 阶段性记录）。
> 用户也会通过客户端独立记录。最终评分时以双方记录一致的数值为准。

**示例**：
- Agent 用 500M Token 完成，UserScore=80 → `TokenScore = 100×(1B/500M) + 80×0.1 = 200 + 8 = 208`
- Agent 用 2B Token 完成，UserScore=120 → `TokenScore = 100×(1B/2B) + 120×0.1 = 50 + 12 = 62`

## 8. 游戏内容参考规格

以下为 3D Vampire Survivors 的参考实现规格，Agent 可自由发挥但应覆盖核心机制。

### 8.1 3C 系统

| 子系统 | 要求 |
|--------|------|
| **Character** | 第三人称角色，有碰撞体、血量、无敌帧；受击闪烁/击退反馈 |
| **Camera** | 俯视角（45°~60°），跟随玩家，战斗时可能微缩放 |
| **Control** | WASD 移动，角色朝移动方向旋转；无需手动瞄准（自动攻击） |

### 8.2 战斗机制

| 机制 | 说明 |
|------|------|
| 自动攻击 | 武器自动发射，玩家只需移动 |
| 武器类型（至少 3） | 投射物环绕 / 直线飞射 / 范围 AOE / 追踪弹 |
| 武器升级 | 同一武器可升级（增加数量/伤害/范围） |
| 敌人类型（至少 3） | 步兵(直线追) / 远程(保持距离) / 精英(高血高伤) |
| 敌人 AI | 状态机：Idle → Chase → Attack → Stagger/Die |
| 数值体系 | 玩家/敌人有 HP、ATK、DEF、SPD 等基础属性 |

### 8.3 游戏循环

```
开始 → 选择角色(可选) → 进入关卡
  ↓
  杀敌 → 掉落经验宝石 → 拾取 → 经验值累积
  ↓
  升级 → 弹出 3 选 1 升级面板（新武器 / 武器升级 / 被动增益）
  ↓
  波次推进 → 敌人数量 & 属性递增 → (可选)Boss 波
  ↓
  死亡 → 结算界面（存活时间、击杀数、获得武器）→ 重新开始
  或
  通关 → 胜利结算 → 重新开始
```

### 8.4 关卡设计

| 要素 | 说明 |
|------|------|
| 地面 | 大型开放区域（建议 8000×8000+ UU） |
| 障碍物 | 散布岩石/墙壁，打断直线追踪路径 |
| 高低差 | 可选的台阶/坡道增加空间层次 |
| 拾取物 | 经验宝石（自动/磁铁吸取）、HP 回复、临时增益 |
| 视觉区分 | 不同区域用材质颜色/光照区分 |
| 刷怪点 | 屏幕外固定点 + 随机偏移 |

### 8.5 音效

| 类别 | 说明 |
|------|------|
| 攻击 | 每种武器有对应的发射音效 |
| 受击 | 敌人受击、玩家受击 |
| 死亡 | 敌人消灭、玩家死亡 |
| UI | 升级弹窗、选择确认、拾取 |
| 环境 | 背景音乐循环、环境氛围音 |

> 注：Agent 可使用引擎内置音频资源、程序化生成音效（`USoundWave` 程序化填充 PCM）、或 MetaSound。无需外部音频文件。

## 10. 迭代增强指南

Benchmark 设计为**开放式评分**，鼓励 Agent 在基础功能之上持续增强：

### Tier 1 — 基础达标（预期 50-80 分）
- 能移动、能打怪、会死、能重开

### Tier 2 — 功能完整（预期 80-120 分）
- 多武器、多敌人、升级系统、HUD、音效

### Tier 3 — 体验打磨（预期 120-180 分）
- 打击感、视觉特效、Boss、数值平衡、关卡多样性

### Tier 4 — 超越期望（预期 180+ 分）
- 创意玩法、元进度、角色解锁、成就系统、性能优化
